Two separate paths, same data source. The view graph (build_graph) handles filtered visualization — what you see. The analysis graph handles structured queries — what you ask. Both read from the same Data Store. Query results overlay onto the view graph as highlights/groups/filters.

4. Your Four Queries — Concrete Examples
1. Nodes with more than one MAC
FIND nodes WHERE mac_count > 1 THEN highlight
API Request
{ "target": "nodes", "conditions": [{ "field": "mac_count", "op": ">", "value": 1 }], "action": "highlight" }
NetworkX Implementation
[ip for ip, d in G.nodes(data=True)
if d["mac_count"] > 1]
2. Edges with both DNS and SMB sessions
FIND edges WHERE protocols contains_all [DNS, SMB] THEN highlight
API Request
{ "target": "edges", "conditions": [{ "field": "protocols", "op": "contains_all", "value": ["DNS", "SMB"] }], "action": "highlight" }
NetworkX Implementation
[(u,v) for u,v,d in G.edges(data=True)
if {"DNS","SMB"} <= d["protocols"]]
3. Group nodes connected ONLY to one server
FIND nodes WHERE connects_only_to = 192.168.1.1 THEN group
API Request
{ "target": "nodes", "conditions": [{ "type": "topology", "op": "connects_only_to", "value": "192.168.1.1" }], "action": "group" }
NetworkX Implementation
[n for n in G.neighbors("192.168.1.1")
if set(G.neighbors(n)) == {"192.168.1.1"}]
4. Highlight nodes with HTTP + specific JA3
FIND nodes WHERE protocols contains HTTP
  AND ja3s contains e7d7...abc THEN highlight
API Request
{ "target": "nodes", "conditions": [ { "field": "protocols", "op": "contains", "value": "HTTP" }, { "field": "ja3s", "op": "contains", "value": "e7d7...abc" } ], "action": "highlight" }
NetworkX Implementation
[ip for ip, d in G.nodes(data=True)
if "HTTP" in d["protocols"]
and "e7d7...abc" in d["ja3s"]]

7. Full Request/Response Flow
POST /api/query
{ "target": "nodes", "conditions": [ { "field": "mac_count", "op": ">", "value": 1 }, { "field": "protocols", "op": "contains", "value": "ARP" } ], "logic": "AND", "action": "highlight" }
Response
{ "matched_nodes": [ { "id": "192.168.1.50", "match_details": { "mac_count": 2, "macs": ["aa:bb:cc:11:22:33", "aa:bb:cc:44:55:66"], "protocols": ["ARP", "DNS"] } }, { "id": "10.0.0.12", "match_details": { ... } } ], "matched_edges": [], "action": "highlight", "total_matched": 2, "total_searched": 48, "summary": "2 of 48 nodes with mac_count > 1 AND protocols contains ARP" }
Frontend receives matched IDs and applies the action as a client-side overlay on the current view graph. The view graph itself is NOT rebuilt. This means queries are instant — no build_graph() call, no re-layout. The graph just re-colors/hides/groups existing nodes.
8. Why NetworkX, Not Neo4j
NetworkX (in-memory) Neo4j (external DB)
Setup pip install (already a dependency) Java runtime + server process + auth
Query latency < 1ms (in-process, no network) 5-50ms (Bolt protocol over TCP)
Memory Python dicts (~2-5x raw data size) JVM heap + page cache + separate from Python
Query language Python list comprehensions (wrapped in API) Cypher (purpose-built for graphs)
Scale ceiling ~1M nodes (fine for pcap analysis) Billions (overkill for single-capture)
Persistence In-memory only (lost on restart) Disk-backed (survives restart)
Deployment Zero config, bundled with SwiftEye Separate service, docker-compose or install
User friction None — drag pcap, query immediately "Install Neo4j first" in README
When Neo4j WOULD make sense: If SwiftEye grows to support persistent investigations across multiple captures, correlation between captures over time, or multi-user shared analysis — that's when a real graph DB pays off. For single-capture, single-researcher analysis, NetworkX gives you the same query power with zero infrastructure.

☐ Topology operators (connects_to, degree, etc.)
☐ Actions: group, hide, isolate
☐ AND/OR logic combinator
☐ Query history / saved queries
☐ Result detail panel (expand matched nodes)
☐ Export matched set as CSV
Phase 3 — Power User
☐ Raw query textbox (simple DSL or Python subset)
☐ Compound queries (pipe result of Q1 into Q2)
☐ Time-scoped queries ("in the last 5 minutes of capture")
☐ Regex matching on hostnames/queries
☐ Plugin-provided query fields (OS, risk scores)
☐ Query-based alerting ("flag if mac_count > 1")
