Key difference from your diagram: The backend doesn't have a persistent "Graph Instance." It has a Data Store (packets + sessions + plugin results) and a stateless graph builder that recomputes on every request. Algorithms (clustering, pathfinding) are post-processing on the computed graph, not inputs to it. The DB replaces the storage layer underneath — filter_packets() becomes a SQL query, but the pipeline shape stays the same.
On the arrows:
Ingestion → Store: List[PacketRecord] (one-time on upload)
Store → build_graph(): filtered packets + entity_map + hostname_map (per request)
build_graph() → API: { nodes: [...], edges: [...], clusters: {...} } (JSON)
API → Frontend: same JSON, stored as rawGraph
rawGraph → visibleNodes/Edges: client-side transforms (useMemo, no mutation)
User controls → API: query params (time_start, protocols, subnet_prefix, etc.)

Your Model vs Current Reality

Your Diagram Current Reality Implication
Backend Graph Instance No persistent graph. Data Store + stateless build_graph() Adding DB replaces the store, not the graph. Pipeline shape unchanged.
DB → Graph DB replaces in-memory lists inside Data Store DB is under the store, not beside it. filter_packets() becomes SQL.
Algorithms → Graph Algorithms run ON the computed graph (post-processing) Clustering/pathfinding are transforms, not data sources.
Sessions → Graph Sessions are built once on upload, stored alongside packets Sessions feed build_graph, but they're peers of packets, not a separate input.
Frontend Graph Instance rawGraph → view transforms → visibleNodes/Edges → D3 Closer to your model, but it's a pipeline, not a hub.
Filter → Frontend Graph Some filters refetch (API), some are client-side (no refetch) Important distinction: display filter is instant, protocol toggle hits the backend.
